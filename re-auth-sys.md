# ReManageSociety Authentication System

## Status

Approved design for the next authentication implementation. This is a simple, fast, password-based system. MFA, OTP, and separate authentication systems per role are not part of this version.

## Core model

Authentication, society membership, and permissions are separate concerns.

```text
Authentication: email + password -> authenticated session
Membership:     user belongs to a society
Authorization:  membership role -> allowed actions within that society
```

All account types use the same login endpoint and the same email/password session mechanism. Roles only control what a user can access after login.

## Roles and permission mapping

| Product role | Internal permission role | Access scope |
| --- | --- | --- |
| Chairman | `society_admin` | Full control of their own society only |
| Secretary | `committee` | Operations, residents, occupancy, notices, and helpdesk |
| Treasurer | `treasurer` | Finance and audit only |
| Guard | `guard` | Gate, visitor, and security operations |
| Flat owner/member | `member` | Their own flat and resident features |
| Tenant | `tenant` | Their own flat and tenant features |

No role may read or modify data from another society.

## Chairman self-service registration

New people can register and become Chairman by creating a society. This is the chosen bootstrap model.

```text
Register -> create society -> become Chairman -> enter dashboard
```

The registration flow collects, at minimum:

- Chairman name
- Email
- Password
- Phone number
- Society name
- Society address or location

The server must create the following records atomically. If any step fails, none of the records are saved.

```text
User
  email, passwordHash, active

Society
  name, address, joinCode

SocietyMembership
  userId, societyId, role = chairman / society_admin, status = active
```

The new Chairman can enter the dashboard immediately, complete society setup, create staff accounts, and share the resident join code. A society may have a setup status such as `new`, `setup_incomplete`, or `operational`; this must not prevent the first Chairman from accessing their society.

## Lifecycle

1. A Chairman registers with email and password, creates a society, and is assigned `society_admin` for that society.
2. The Chairman creates Secretary, Treasurer, and Guard accounts for that same society.
3. Each staff member logs in through the normal email/password login.
4. The Chairman shares the society join code with residents.
5. An owner or tenant signs up with email and password, then enters the society join code.
6. The resident completes their profile and selects an existing flat or requests registration of a flat.
7. The resident declares their occupancy type: Owner or Tenant.
8. The system validates flat occupancy and grants final `member` or `tenant` access.

## Staff account creation

Chairman is the only role that creates staff accounts in this first version.

```text
Chairman in Society A
  -> creates Secretary, Treasurer, or Guard
  -> server assigns Society A automatically
  -> staff member signs in with normal email + password
  -> role policy restricts access to Society A
```

The server, not the browser, determines the society assignment. A Chairman must never be able to create staff for another society.

## Owner and tenant onboarding

```text
Email + password
  -> enter society join code
  -> society membership is created as pending
  -> select an existing flat or request a new flat
  -> declare Owner or Tenant
  -> Chairman or Secretary approves if verification is required
  -> user receives final member or tenant permissions
```

A join code proves that the user may begin joining that society. It does not prove ownership or tenancy of any flat. Flat claims require validation before full access is granted.

## Account and security rules

- Every person has exactly one `User` account with a globally unique email.
- Passwords are stored only as secure password hashes; plaintext passwords are never stored or logged.
- All staff accounts must have a non-null `societyId` or equivalent society membership.
- A Chairman can create staff only for their own `societyId`.
- A society starts with exactly one active Chairman.
- One email can create only one society in the first version.
- The join code is generated server-side and can be regenerated by the Chairman.
- Residents may join using a society code, but cannot create staff accounts.
- The browser must never decide or be trusted to assign a privileged role. The server assigns `society_admin` during society creation and validates all other role changes.
- Server-side permission checks must protect every route and API action. Hiding a sidebar item is not authorization.
- Multi-society memberships are explicitly out of scope for the first version. The model can be extended later if needed.
- Password reset should later use a standard short-lived reset token or link. For the initial version, Chairman can reset staff passwords through Team & Roles.

## Deliberately out of scope for the first version

- MFA, OTP, magic links, and social login
- Separate authentication systems for Secretary, Treasurer, or Guard
- Multi-society membership for one user
- Automated ownership verification from external records

## Implementation direction

Implement one standard email/password authentication flow and a society-scoped authorization layer. Account provisioning differs by actor (self-service Chairman registration, Chairman-created staff, and join-code resident onboarding), but all successful users authenticate through the same login and receive a session containing their identity, active society, and membership role.
